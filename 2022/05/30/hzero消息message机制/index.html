hzero消息message机制

hzero
发布日期:   2022-05-30

项目上报错:

意思是找不到这个 messageCode，这个 code 是依赖的产品的，经检查，messageCode 是存在于产品 jar 的 message_zh_CN.properties 文件中的。
最后经过调试，发现是项目本地也有一个 message_zh_CN.properties，和 jar 包中的冲突了，导致找不到这个 messageCode。排查很简单，但也引来了其他的疑问
为什么其他的项目没有出现该问题？
这个文件产品一直采用的是这个名字，其他 hzero 的产品是怎么解决这个问题的？

第一个问题很简单，其他项目直接把该文件中的内容复制出来了。
第二个问题比较引人深思，但看看源码就大概知道，比如拿主键加密的组件来看

改了名字并且将其加入到 Basenames 中。
于是我尝试在当前项目中获取该组件中的 message，可惜的是获取不到，报错了

而且 messageSource 中获取的 Basenames 中并没有上面添加的 message

那组件中的 MessageAccessor 调用 addBeannames 的作用就很疑惑了。而且其他的 message 是如何添加进去的，比如 message_redis。
试了下其他产品的另一种写法：

@Bean  
public SmartInitializingSingleton hbgtMessagesLoader(HierarchicalMessageSource hierarchicalMessageSource) {  
    return () -> {  
        String messagePath = "classpath:messages/messages_hbgt";  
        // 加入到hzero的MessageAccessor中  
        MessageAccessor.addBasenames(messagePath);  
        // 加入到spring boot的MessageSource中  
        MessageSource parentMessageSource;  
        if (hierarchicalMessageSource != null && (parentMessageSource = hierarchicalMessageSource.getParentMessageSource()) != null && parentMessageSource instanceof ReloadableResourceBundleMessageSource) {  
        // 获取到spring中管理的messageSource，并将该message添加进去
            ((ReloadableResourceBundleMessageSource) parentMessageSource).addBasenames(messagePath);  
        }  
    };  
  
}

采用这种写法就可以成功获取到了。

从上面注释可以看出，这个类本质上是一个工具类，主要是为了提供一个直接通过 messageCode 来获取 message 的机制，而不需要手动注入 messageSource，虽然其本质还是通过 messageSource。把它看作一个 Utils 就可以了。

MessageSource

但是我们日常使用基本都是通过注入的 MessageSource 来获取 message

就不详细说了，应该是会将 Message 缓存到 redis 中。

个人觉得这个有点多余，不知道是为了什么考虑，本身 messageCode 解析也是在本地做的，为什么还要去 redis 中获取，虽然不用解析但多了一次网络 io，而且 springboot 默认的机制也是自带本地缓存的，并不是每一次获取都要解析。

注入的地方

@Bean  
@ConditionalOnMissingBean({RedisMessageSource.class})  
public MessageSource messageSource(RedisHelper redisHelper) {  
    ReloadableResourceBundleMessageSource parentMessageSource = new ReloadableResourceBundleMessageSource();  
    parentMessageSource.setBasenames(MessageAccessor.getBasenames());  
    parentMessageSource.setDefaultEncoding(Charsets.UTF_8.displayName());  
    RedisMessageSource messageSource = new RedisMessageSource(redisHelper);  
    messageSource.setParentMessageSource(parentMessageSource);  
    return messageSource;  
}

可以看到，这里也是 new 了一个父 MessageSource 来作为 RedisMessageSource 的 parent。

我在代码中使用的 messageSource 就是在这里注入的。

所以仅仅 MessageAccessor.addBasenames(new String\[\]{"classpath: messages/messages_keyencrypt"}); 对这里的 messageSource 是没有任何影响的，取出来的 basenames 中自然是没有 keyencrypt 的 properties，他是加入到了 MessageAccessor 中的 messageSource（注意和这里注入的 messageSource 是两个单独的对象）。

可以看到，甚至还有其他组件的 message 文件，所以通过 MessageAccessor 就能够直接获取的

总结

Hzero 设计 MessageAccessor 的初衷应该就是为了统一消息 message 的获取方式，将其作为了一个工具类来使用，该类存在于 core 中

所以在其他的一些 hzero 的组件中，基本都是使用的这个 MessageAccessor，所以也会将其对应的 message 加入到该 messageAccessor 中的 messageSource 中，一般都是这种写法：

@Bean  
public SmartInitializingSingleton keyEncryptMessagesLoader() {  
    return () -> {  
        MessageAccessor.addBasenames(new String[]{"classpath:messages/messages_keyencrypt"});  
    };  
}

但是对于我们来说，一般还是使用 MessageSource 比较多，弊端就是无法使用自定义的 message，除非手动添加进去，比如下面的写法：

@Bean  
public SmartInitializingSingleton hbgtMessagesLoader(HierarchicalMessageSource hierarchicalMessageSource) {  
    return () -> {  
        String messagePath = "classpath:messages/messages_hbgt";  
        // 加入到spring boot的MessageSource中  
        MessageSource parentMessageSource;  
        if (hierarchicalMessageSource != null && (parentMessageSource = hierarchicalMessageSource.getParentMessageSource()) != null && parentMessageSource instanceof ReloadableResourceBundleMessageSource) {  
            ((ReloadableResourceBundleMessageSource) parentMessageSource).addBasenames(messagePath);  
        }  
    };  
  
}

当然，我们也是可以使用 MessageAccessor，因为它就是设计出来作为工具类来使用的。
